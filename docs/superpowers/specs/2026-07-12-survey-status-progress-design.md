# Статусы и прогресс опросов в киоск-списке сотрудника

**Дата:** 2026-07-12
**Статус:** дизайн согласован, ожидает финального ревью спецификации

## 1. Проблема и цель

На экране `/scan` после входа (лицо + SMS-код) сотрудник видит список опросов
(«Mavjud so‘rovlar»). Сейчас список:

- **дублирует опросы:** начатый-но-не-сданный опрос показывается **дважды** — в
  секции «Davom ettirish» (продолжить) и в секции «Mavjud so‘rovlar» (доступные).
  На скринах пользователя «Глубокий опрос» виден в обеих секциях.
- **не показывает статус:** нельзя понять, начат опрос или нет, на какой стадии
  прохождение, сколько осталось.
- **молча прячет** завершённые и просроченные — они просто исчезают.

**Причина дубля (бэкенд):** `due_surveys()`
(`backend/apps/surveys/scheduling.py:29-62`) исключает из выдачи только
**завершённые** опросы (`_completed_since` / `_completed_ever`). Начатый опрос
`completed_at IS NULL` → он остаётся в `due` (available), и одновременно
попадает в `in-progress` (continue). Отсюда две строки на один опрос.

**Цель:** сделать список «полноценным» — один опрос = одна строка с понятным
**статусом** и, для начатых, **прогрессом** (`X / Y` вопросов). Логику статуса
считает **бэкенд** (единый источник правды), фронт только рисует бейджи.

## 2. Согласованные решения (ответы пользователя)

- **Объём списка:** показываем только **активные** опросы — «Не начат» и «В
  процессе», каждый со своим бейджем. Завершённые и просроченные в списке **не
  показываем**.
- **Просрочка = read-only:** опрос с истёкшим окном пройти/сдать **нельзя**
  (бэкенд блокирует `start`/`submit`).
- **Прогресс:** для «В процессе» — **полоса прогресса + `X / Y` вопросов**.

## 3. Модель данных (что уже есть — без изменений схемы)

Новых таблиц/полей **не требуется**. Всё вычисляется из существующего:

- `Test` (`backend/apps/surveys/models.py:10-42`) — расписание:
  - **Режим A** «после найма»: `is_after_application=True`, `after_days=N` —
    разовый, дедлайна нет (висит, пока не сдан).
  - **Режим B** «периодический»: `month` (список месяцев; `[]` = каждый месяц) +
    окно дней месяца `test_days_from..test_days_to` (`:25-27`, целые 1..31, это
    **не даты**, а дни месяца). У режима B есть «конец окна» → отсюда «срок истёк».
- `SurveySession` (`:125-170`) — истина в двух датах: `started_at` (всегда),
  `completed_at` (NULL до сдачи). `status` — вычисляемое `@property` (`:158-165`):
  `completed` / `abandoned` (старше `SURVEY_SESSION_ABANDONED_AFTER_HOURS`, деф.
  24 ч) / `in_progress`.
- `Answer` (`:173-189`) — по строке на предъявленный вопрос (создаётся пустой в
  `start_survey_session`, заполняется автосейвом). Поля ответа:
  `selected_option_ids: JSONField(list)` и `text_value: TextField`.

**Определение «отвечено» (метрика прогресса):** вопрос считается отвеченным,
если у его `Answer` **`selected_option_ids` непустой список** ИЛИ
**`text_value.strip()` непустой**. Тогда:
- `total_count` = число `Answer` у сессии (`session.answers.count()`),
- `answered_count` = число «отвеченных» по правилу выше.

## 4. Определение статуса (бэкенд, на текущий день `today`)

Для пары (сотрудник, тест) в текущем периоде:

| Статус | Условие | В списке |
|---|---|---|
| **completed** | завершён в текущем периоде (`_completed_since`/`_completed_ever`) | не показываем |
| **in_progress** | есть «живая» сессия: `completed_at IS NULL`, не заброшена (в пределах 24 ч), **и окно теста ещё открыто** | секция «Davom ettirish» |
| **not_started** | тест сейчас due, живой сессии нет | секция «Mavjud» |
| **expired** | окно закрылось, не завершён (только режим B) | не показываем; `start`/`submit` запрещены |

«Окно открыто сейчас» для режима B = `today.month ∈ month` и
`test_days_from ≤ today.day ≤ test_days_to` (верх зажат последним днём месяца) —
та же логика, что уже в `due_surveys` (`scheduling.py:48-59`). Для режима A окно
считается всегда открытым (дедлайна нет).

## 5. Изменения бэкенда

### 5.1 `due` = только реально не начатые (убираем дубль)
`due_surveys(employee, today)` (`scheduling.py:29-62`) дополнительно **исключает**
тесты, по которым у сотрудника есть **живая сессия** (в процессе). Тогда `due` =
строго «Не начат», `in-progress` = «В процессе», пересечений нет.

Реализация: переиспользуем `_live_session(employee, test)`
(`backend/apps/surveys/services.py:126-137`); тест добавляется в результат, только
если живой сессии нет. Список тестов небольшой — цена по-тестовой проверки
приемлема; при желании оптимизируем одним запросом «тесты с живой сессией».

### 5.2 Прогресс в `in-progress`
- `in_progress_sessions(employee)` (`services.py:140-149`) — добавить
  `.prefetch_related("answers")` (для подсчёта без N+1) и **исключить сессии с
  истёкшим окном** теста (они «expired», а не «активные»; по решению п.2 их не
  показываем и не даём сдавать).
- `SurveySessionSerializer` (`backend/apps/surveys/serializers.py:178-198`) —
  добавить два read-only поля через `SerializerMethodField`:
  `answered_count`, `total_count` (считаются по правилу из п.3, по
  предзагруженным `answers`).

### 5.3 Guard просрочки (read-only)
- Новый помощник в `scheduling.py`, напр. `is_window_open(test, today) -> bool`
  (для режима A всегда `True`; для B — по окну месяца/дней). Выносим общий расчёт
  окна, чтобы не дублировать логику `due_surveys`.
- `start` (`views.py:370-399` → `start_survey_session`): если у теста окно
  закрыто (и это не возобновление уже живой сессии) → отклонить как
  `survey_expired` (HTTP 400/409, тело `{"detail": ..., "code": "survey_expired"}`).
- `submit` (`{id}/submit/`): если окно теста закрылось к моменту сдачи → отклонить
  `survey_expired`.
- **Явное следствие (зафиксировано осознанно):** сотрудник, начавший опрос, но не
  успевший до конца окна, теряет возможность досдать — опрос исчезает из списка и
  сдать его нельзя. Это прямое следствие выбора «только активные» + «просрочка =
  read-only». Если позже захотим показывать серую строку «Срок истёк» — это
  отдельное небольшое расширение (см. п.9).

### 5.4 Тесты (pytest)
- `due` не содержит теста с живой сессией (нет дубля).
- `in-progress` отдаёт `answered_count`/`total_count`; счёт растёт после автосейва;
  пустой ответ не считается; выбор опций и текст считаются.
- `in-progress` исключает сессию с истёкшим окном.
- `is_window_open`: режим A всегда открыт; режим B — граничные дни (`from`, `to`,
  `to+1`), неверный месяц, зажим по последнему дню месяца.
- `start`/`submit` истёкшего теста → `survey_expired`; due-тест стартует нормально;
  возобновление живой сессии внутри окна работает.

## 8. Критерии готовности

1. Один опрос — одна строка; дубля «в процессе» + «доступен» больше нет.
2. Не начатый опрос помечен бейджем «Не начат»; начатый — «В процессе» + полоса
   `X / Y вопросов`, где `X` растёт по мере автосейва.
3. Завершённые и просроченные в списке не показываются.
4. Опрос с истёкшим окном нельзя открыть/сдать (`survey_expired`); фронт
   показывает понятный тост.
5. Прогресс считается корректно (пустой ответ не засчитан; опции и текст —
   засчитаны).
6. Все новые тесты бэкенда зелёные; фронт `typecheck` + `build` проходят.
