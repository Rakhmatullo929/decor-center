# Locomotive Decort — Knowledge Assessment & Medical Examination System

Web application for the Bukhara Region Locomotive Decort ("Uzbekistan Railways" JSC):
professional knowledge tests with Face ID and TTS, occupational-safety tests, and daily
medical examination records.

- Spec: [Technical_Specification_Locomotive_Decort_EN.md](Technical_Specification_Locomotive_Decort_EN.md)
- Approved stage-1 design: [docs/superpowers/specs/2026-06-11-backend-foundation-design.md](docs/superpowers/specs/2026-06-11-backend-foundation-design.md)

## Structure

| Path | Description |
|---|---|
| `backend/` | Django 5.2 + DRF API (PostgreSQL, JWT, role-based access) |
| `frontend/` | React 18 + TypeScript SPA (MUI, React Query, RHF, i18n uz/ru) |
| `frontend/cursor/rules/` | Frontend coding conventions (API layering, URL state, RBAC, i18n) |
| `docker-compose.yml` | PostgreSQL for local development |
| `Makefile` | Developer task runner — `make help` for the full list |

## Quick start (Makefile)

```bash
make setup    # .env files, deps, Postgres (docker), migrate, seed accounts
make dev      # run backend (:8000) + frontend (:3000) together; Ctrl-C stops both
```

`make help` lists every target (run, test, lint, typecheck, docker, …). The sections
below describe the same steps run by hand.

## Backend — quick start

```bash
cd backend
python3 -m venv .venv
.venv/bin/pip install -r requirements/dev.txt

# PostgreSQL: either local `createdb decor` or `docker compose up -d db` from the repo root
cp .env.example .env                      # adjust DATABASE_URL if needed

.venv/bin/python manage.py migrate
.venv/bin/python manage.py seed_initial_data   # 6 specialties + admin/medic/specialist accounts
.venv/bin/python manage.py runserver
```

- API docs (Swagger): http://127.0.0.1:8000/api/v1/docs/
- Django admin: http://127.0.0.1:8000/admin/
- Seed account passwords come from `DECOR_*_PASSWORD` env vars (see `.env.example`); change them in production.

### Tests & lint

```bash
cd backend
.venv/bin/python -m pytest
.venv/bin/ruff check .
```

### Key endpoints (`/api/v1/`)

| Endpoint | Purpose | Roles |
|---|---|---|
| `auth/login/`, `auth/refresh/`, `auth/me/` | JWT auth | all |
| `specialties/`, `employees/` | directory management, search | read: all; write: admin |
| `instructions/` + `{id}/generate/` | upload instructions, trigger AI question generation | admin |
| `questions/` + `{id}/approve/` | test bank management | admin |
| `test-sessions/start/`, `test-sessions/{id}/submit/` | test flow with Face ID gate | specialist, admin |
| `test-sessions/` | results browsing | admin |
| `medical-checks/` | medical examination records + history filters | medic, admin |

### Integrations (mocked at this stage)

Face recognition and AI test generation are behind service interfaces
(`backend/apps/integrations/`), selected via `DECOR_*_BACKEND` env vars with
deterministic mocks until real providers are wired in (phase 4+). Question TTS is
generated by a `post_save` signal on `Question` calling UzbekVoice directly
(`UZBEKVOICE_API_KEY`, `DECOR_TTS_VOICE_UZ`); set `DECOR_TTS_ASYNC=false` to run it inline.

## Frontend — quick start

```bash
cd frontend
npm install
cp .env.example .env        # REACT_APP_HOST_API=http://localhost:8000
npm start                   # http://localhost:3000 (backend must be running)
```

Role screens (login with the seed accounts):

| Account | Screens |
|---|---|
| `admin` | Employees, Specialties, Question bank (+approve), Instructions (+AI generate), Results (+answer breakdown), Medical (audited edit) |
| `medic` | Medical examination journal + entry form with out-of-range warnings |
| `specialist` | Kiosk test flow: module → employee (photo search) → Face ID camera → 10 questions (TTS for module 1) → result |

Checks: `npx tsc --noEmit`, `npm run lint`, `npm run build`.
Camera note: `getUserMedia` works on `localhost` or HTTPS only (SRS §11.2).
